## 一、需求背景

### 1.1 现状分析

Git Worktree Manager 已具备 **Diff 对比** 功能，可以清晰展示当前分支与主分支的代码差异。用户在提交代码前，需要：

1. 查看 Diff 了解变更内容
2. 人工检查是否有潜在问题
3. 思考是否有遗漏或优化空间

### 1.2 用户痛点

| 场景 | 痛点 |
|------|------|
| **代码审查** | 自己很难发现自己的问题 |
| **团队协作** | PR Review 周期长，反馈延迟 |
| **新手开发者** | 缺乏代码质量意识，不知道如何改进 |
| **紧急修复** | 时间紧迫，容易遗漏问题 |

### 1.3 解决方案

在现有 Diff 面板基础上，增加 **AI 智能评审** 按钮，一键获取 AI 对代码变更的评审意见。

### 2.2 功能清单

#### P0 - MVP 功能

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| **AI 评审按钮** | 在 Diff 面板顶部添加按钮 | P0 |
| **评审结果展示** | 分类展示：潜在问题 / 改进建议 / 亮点 | P0 |
| **代码定位** | 点击评审项跳转到对应代码行 | P0 |
| **评审历史** | 缓存评审结果，避免重复调用 | P0 |

#### P1 - 增强功能

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| **多模型支持** | 支持 OpenAI / Claude / 本地模型 | P1 |
| **自定义提示词** | 用户可自定义评审关注点 | P1 |
| **评审模式** | 快速评审 / 深度评审 | P1 |
| **导出报告** | 导出 Markdown 格式评审报告 | P1 |

#### P2 - 高级功能

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| **增量评审** | 只评审最近 N 次提交 | P2 |
| **团队规则** | 自定义团队代码规范作为评审依据 | P2 |
| **评审统计** | 统计常见问题类型 | P2 |

## 三、技术方案

### 3.1 API 选型

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **OpenAI API** | 效果好，生态成熟 | 付费，隐私担忧 | ✅ 首选 |
| **Claude API** | 代码理解能力强 | 付费 | ✅ 备选 |
| **本地模型 (Ollama)** | 隐私安全，免费 | 效果一般，资源占用高 | 🔶 可选 |
| **自定义端点** | 企业可自建 | 需要运维 | 🔶 可选 |

### 3.2 数据流设计

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Diff 内容   │ ──▶ │  Prompt 构建  │ ──▶ │   AI API     │
│  (本地获取)  │     │  (本地处理)   │     │  (远程/本地) │
└──────────────┘     └──────────────┘     └──────────────┘
                                                  │
                                                  ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  UI 渲染     │ ◀── │  结果解析    │ ◀── │  AI Response │
│              │     │  (结构化)    │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 3.3 Prompt 设计

```markdown
你是一位资深代码审查专家。请对以下代码变更进行评审。

## 评审规则
1. 关注代码质量、安全性、性能、可维护性
2. 给出具体的改进建议，而非泛泛而谈
3. 用中文回复

## 代码变更
\`\`\`diff
{diff_content}
\`\`\`

## 输出格式
请按以下 JSON 格式输出：
{
  "issues": [
    {
      "severity": "warning" | "error",
      "file": "文件路径",
      "line": 行号,
      "message": "问题描述",
      "suggestion": "改进建议"
    }
  ],
  "improvements": ["改进建议1", "改进建议2"],
  "highlights": ["亮点1"]
}
```

### 3.4 隐私保护

| 措施 | 说明 |
|------|------|
| **本地预处理** | 脱敏敏感信息后再发送 |
| **用户确认** | 首次使用提示隐私说明 |
| **本地模型选项** | 支持完全本地化部署 |
| **不存储代码** | 只发送 Diff，不存储 |

### 3.5 成本控制

| 措施 | 说明 |
|------|------|
| **结果缓存** | 相同 Diff 不重复调用 |
| **模型选择** | 用户可选择更便宜的模型 |
| **限制频率** | 每日调用次数限制 |
| **Token 控制** | 只发送必要的 Diff 内容 |

## 五、用户故事

### US-001: 首次使用 AI 评审

```
作为一个 开发者
我想要 在提交代码前获得 AI 的评审意见
以便于 发现潜在问题并改进代码质量

验收标准:
- [ ] Diff 面板有明显的 AI 评审按钮
- [ ] 首次点击弹出 API Key 配置引导
- [ ] 配置完成后自动开始评审
- [ ] 评审结果清晰分类展示
```

### US-002: 查看评审结果

```
作为一个 开发者
我想要 快速定位评审中提到的问题代码
以便于 理解问题并进行修改

验收标准:
- [ ] 点击评审项跳转到对应代码行
- [ ] 问题按严重程度排序
- [ ] 支持忽略某条评审意见
- [ ] 支持复制建议内容
```

### US-003: 配置 AI 提供商

```
作为一个 开发者
我想要 选择不同的 AI 提供商
以便于 根据需求选择最合适的模型

验收标准:
- [ ] 支持 OpenAI / Claude / 本地模型
- [ ] 支持自定义 API 端点
- [ ] 配置持久化存储
- [ ] 提供测试连接功能
```

## 六、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| API Key 泄露 | 低 | 高 | 本地加密存储，不明文显示 |
| AI 返回质量不稳定 | 中 | 中 | 优化 Prompt，允许重新评审 |
| 调用成本过高 | 中 | 中 | 结果缓存，Token 控制 |
| 用户隐私担忧 | 中 | 中 | 支持本地模型，明确隐私说明 |
| API 不可用 | 低 | 中 | 多提供商支持，本地备选 |
