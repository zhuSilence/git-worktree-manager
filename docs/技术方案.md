## 一、技术选型

### 1.1 技术栈总览

```
┌─────────────────────────────────────────────────────────────┐
│                     前端 (UI Layer)                          │
│  React 18 + TypeScript + TailwindCSS + Radix UI             │
│  状态: Zustand    i18n: i18next    图标: lucide-react       │
├─────────────────────────────────────────────────────────────┤
│                    桌面应用框架                               │
│                   Tauri 2.0 (Rust)                          │
├─────────────────────────────────────────────────────────────┤
│                    后端 (Core Layer)                         │
│          Rust + git2 (libgit2 bindings)                     │
├─────────────────────────────────────────────────────────────┤
│                    系统 API                                  │
│        文件系统 / 进程管理 / Shell 集成                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 技术选型理由

| 技术 | 选型理由 |
|------|----------|
| **Tauri 2.0** | 比 Electron 小 10x，内存占用低，安全性好，支持跨平台 |
| **React 18** | 生态成熟，组件化开发，支持并发渲染 |
| **TypeScript** | 类型安全，IDE 支持好 |
| **TailwindCSS** | 原子化 CSS，开发效率高 |
| **Radix UI** | 无样式原语组件，可访问性好 |
| **Rust (后端)** | 性能极高，内存安全，与 Tauri 无缝集成 |
| **git2** | Rust 的 libgit2 绑定，直接操作 Git 仓库 |

## 三、IPC 合约

### 3.1 命令命名规则

前端 `invoke('command_name_cmd', { camelCaseParams })` 映射到 Rust `#[command] pub async fn command_name_cmd(snake_case_params)`。

Tauri 自动转换 camelCase (JS) ↔ snake_case (Rust)。

### 3.2 命令注册

所有命令在 `lib.rs` 的 `generate_handler![]` 中注册。

### 3.3 命令分类

| 分类 | 命令文件 | 核心命令 |
|------|----------|----------|
| Worktree 管理 | `commands/worktree.rs` | `list_worktrees_cmd`, `create_worktree_cmd`, `delete_worktree_cmd`, `get_worktree_status_cmd`, `get_worktree_diff_cmd`, `get_timeline_cmd` |
| 分支操作 | `commands/worktree.rs` | `list_branches_cmd`, `switch_branch_cmd`, `create_branch_cmd`, `push_branch_cmd`, `pull_branch_cmd`, `fetch_remote_cmd` |
| 合并 | `commands/merge.rs` | `merge_branch_cmd`, `abort_merge_cmd`, `complete_merge_cmd`, `check_merge_conflicts_cmd`, `merge_from_main_cmd` |
| Hotfix | `commands/worktree.rs` | `start_hotfix_cmd`, `finish_hotfix_cmd`, `abort_hotfix_cmd` |
| AI | `commands/ai_review.rs` | `ai_review_cmd`, `ai_naming_suggestion_cmd`, `test_ai_connection_cmd` |
| 操作日志 | `commands/log.rs` | `list_operation_logs_cmd`, `export_operation_logs_cmd`, `cleanup_old_logs_cmd` |
| 备份/保护 | `commands/log.rs` | `check_delete_protection_cmd`, `create_backup_cmd`, `restore_backup_cmd`, `list_backups_cmd`, `delete_backup_cmd` |
| 冲突检测 | `commands/worktree.rs` | `detect_worktree_conflicts_cmd`, `get_three_way_diff_cmd` |
| 编辑器 | `commands/worktree.rs` | `open_in_ide_cmd`, `open_in_terminal_cmd`, `open_in_file_manager_cmd` |

## 四、核心数据模型

### 4.1 前端类型定义（TypeScript）

核心枚举：

| 类型 | 值 |
|------|-----|
| WorktreeStatus | Clean / Dirty / Unpushed / Conflicted / Detached / Unknown |
| OperationType | Create / Delete / Switch / Prune / BatchDelete |
| OperationResult | Success / Failed |

核心接口：`Worktree`, `RepositoryInfo`, `BranchInfo`, `LastCommit`, `SyncStatus`, `MergeParams`, `MergeResult`, `HotfixInfo`, `ThreeWayDiff`, `AIConfig`, `AIReviewResult`, `OperationLog`, `OperationLogFilter`, `BackupInfo`, `WorktreeGroup`, `AppSettings`

### 4.2 后端数据模型（Rust）

Rust 模型使用 serde 序列化，字段名使用 snake_case，通过 Tauri IPC 自动转为前端 camelCase。

### 4.3 持久化存储

| 数据 | 存储方式 | 位置 |
|------|----------|------|
| Worktree 元数据 | localStorage | 浏览器 |
| 标注/备注 | localStorage | 浏览器 |
| 分组配置 | localStorage | 浏览器 |
| AI 配置 | localStorage | 浏览器 |
| 应用设置 | localStorage (Zustand persist) | 浏览器 |
| 操作日志 | JSONL 文件 | `~/.worktree-manager/logs/operations.jsonl` |
| 备份 | Git stash + 元数据 | `~/.worktree-manager/backups/` |

## 五、状态管理架构

### 5.1 Zustand Store 划分

| Store | 职责 | 持久化 |
|-------|------|--------|
| worktreeStore | Worktree CRUD、加载状态、事务模式 | 否 |
| repositoryStore | 多仓库管理、仓库有效性检测 | 是 |
| settingsStore | 用户偏好（IDE/终端/主题/语言） | 是 |
| aiReviewStore | AI 评审状态和结果 | 否 |
| groupsStore | 分组管理 | 是 |
| updateStore | 自动更新状态 | 否 |

### 5.2 事务模式

worktreeStore 使用乐观更新 + 回滚模式：

1. 先更新本地状态（乐观）
2. 调用后端 IPC
3. 成功：刷新列表
4. 失败：回滚本地状态

## 六、安全设计

### 6.1 安全原则

1. **最小权限原则**：Tauri capabilities 仅授权必要权限
2. **输入验证**：所有路径参数经过 `validate_path` 验证
3. **分支名清理**：`sanitize_branch_name` 防止注入
4. **CSP 策略**：严格的内容安全策略

### 6.2 validate_path

- 验证路径存在且规范
- 防止路径遍历攻击（拒绝 `..`、符号链接跳出等）
- 所有接受路径的命令都经过此验证

### 6.3 sanitize_branch_name

- 白名单校验：仅允许 `a-zA-Z0-9_\-/.`
- 拒绝连续斜杠、以 `.` 或 `-` 开头/结尾
- 拒绝 `..` 路径遍历

## 七、自动更新方案

### 7.1 架构

```
应用启动 → 延迟3秒 → tauri-plugin-updater 检查
                               ↓
                    GitHub Releases endpoint
                               ↓
                  对比版本号 → 有更新 → 下载安装
```

### 7.2 配置

- 插件：`tauri-plugin-updater`
- 端点：GitHub Releases JSON
- macOS 产物：`.app.tar.gz`（非 `.dmg`）
- Windows target：`windows`（非 `win`）
- 权限：`updater:default` 需在 `capabilities/default.json` 中声明

### 7.3 CI/CD

GitHub Actions `release.yml` 负责构建和发布，`update-tap.yml` 自动更新 Homebrew Cask。
